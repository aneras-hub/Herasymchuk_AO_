1. Що таке інкапсуляція? Наведіть 5 різних прикладів з вашого коду
Інкапсуляція — це один з основних принципів об’єктно-орієнтованого програмування, який означає приховування внутрішньої реалізації класу та надання доступу до даних через властивості й методи. Завдяки інкапсуляції дані не змінюються напряму, а проходять перевірку через спеціально прописану логіку.
У моєму коді інкапсуляція реалізована в кількох місцях.
Перший приклад — приватне поле FullName у класі Student. Воно не доступне напряму ззовні класу, а змінюється через властивість fullName, де є перевірка на порожнє значення та мінімальну довжину ПІБ.
Другий приклад — поле RecordBookNumber. Воно також є приватним, а доступ до нього здійснюється через властивість recordBookNumber. У setter перевіряється, щоб номер залікової книжки складався рівно з 8 цифр.
Третій приклад — поле AverageGrade. Воно змінюється через властивість averageGrade, яка має приватний setter. Це означає, що напряму ззовні класу середній бал змінити не можна. Для цього використовується метод UpdateAverageGrade() або SetManualGrade().
Четвертий приклад — список студентів _students у класі StudentGroup. Він оголошений як приватний, тому інші частини програми не можуть напряму змінювати список. Для роботи з ним створені методи AddStudent(), RemoveStudent(), FindStudent().
П’ятий приклад — властивість Status у класі Student. Вона має приватний setter, тому статус студента не можна випадково змінити з будь-якого місця програми. Це допомагає краще контролювати стан об’єкта.

2. Порівняйте авто-властивості та властивості з повною реалізацією. У яких випадках ви обирали кожен варіант?
Авто-властивості використовуються тоді, коли для поля не потрібна додаткова логіка перевірки. Вони коротші та зручніші для простого зберігання даних.
У моєму коді прикладами авто-властивостей є:
C#
public DateTime DateOfBirth { get; set; }
public DateTime EnrollmentDate { get; set; }
public string Notes { get; set; } = "Немає нотаток";
public GradeJournal Journal { get; set; } = new();
Ці властивості не потребують складної перевірки під час присвоєння, тому для них достатньо авто-властивостей.
Властивості з повною реалізацією використовуються тоді, коли перед збереженням значення потрібно виконати перевірку або додаткову обробку. Наприклад, у властивості fullName перевіряється, чи не є ПІБ порожнім і чи містить воно мінімум 5 символів. У властивості recordBookNumber перевіряється, чи номер складається з 8 цифр. У властивості personalEmail перевіряється правильність email.
Отже, авто-властивості я використовувала для простих даних, а властивості з повною реалізацією — для важливих полів, які потребують валідації.

3. Для яких даних ви застосували валідацію? Які види валідації використовували?
У програмі валідація застосована для основних даних студента, щоб уникнету неправильного введення.
Для ПІБ використана перевірка на порожній рядок і мінімальну довжину. Якщо користувач вводить порожнє значення або дуже коротке ім’я, програма викидає помилку.
Для номера залікової книжки використана перевірка довжини та числового формату. Номер повинен складатися рівно з 8 цифр.
Для середнього бала використана перевірка діапазону. Значення має бути від 0 до 100. Також застосовується округлення до двох знаків після коми.
Для email використовується клас MailAddress, який перевіряє правильність формату електронної адреси.
Для оцінок у класі GradeJournal також використовується перевірка діапазону. Оцінка не може бути меншою за 0 або більшою за 100.
У головному меню також застосовується перевірка введення дат через DateTime.TryParse, а для пошуку студента за номером використовується int.TryParse .

4. Навіщо потрібні read-only та init-only властивості? Приклади з коду
Read-only властивості потрібні для того, щоб значення можна було тільки отримати, але не змінити напряму. Вони часто використовуються для обчислюваних даних.
У моєму коді прикладом read-only властивості є:
C#
public int Age
{
    get
    {
        var today = DateTime.Today;
        int age = today.Year - DateOfBirth.Year;
        if (DateOfBirth.Date > today.AddYears(-age)) age--;
        return age;
    }
}
Властивість Age не має setter, тому її не можна встановити вручну. Вона автоматично обчислюється на основі дати народження.
Ще один приклад read-only властивості є в класі StudentGroup:
C#
public int GroupSize => _students.Count;
Ця властивість показує кількість студентів у групі, але напряму змінити її не можна.
Init-only властивості потрібні для того, щоб значення можна було задати тільки під час створення об’єкта. У моєму коді повноцінна init-властивість не використана, але використано required для властивості fullName. Це означає, що під час створення об’єкта студент обов’язково повинен отримати ПІБ.
Приклад:
C#
public required string fullName
Це допомагає уникнути створення студента без імені.

5. Поясніть роботу властивості Age. Чому вона обчислювана?
Властивість Age відповідає за визначення віку студента. Вона є обчислюваною, тому що вік не потрібно зберігати окремим числом. Він залежить від дати народження та поточної дати.
У коді береться сьогоднішня дата, від поточного року віднімається рік народження студента. Потім додатково перевіряється, чи був уже день народження у поточному році. Якщо день народження ще не настав, від віку віднімається один рік.
Це правильніше, ніж зберігати вік вручну, тому що вік змінюється щороку. Якщо зберігати його як звичайне поле, його потрібно було б постійно оновлювати. А обчислювана властивість завжди повертає актуальне значення.

6. Як ви реалізували принцип єдиної відповідальності в класах?
Принцип єдиної відповідальності означає, що кожен клас має відповідати тільки за свою частину логіки.
У моїй програмі клас Student відповідає за дані одного студента: ПІБ, дату народження, номер залікової книжки, email, середній бал, статус і нотатки. Також у ньому є методи для перевірки успішності студента, наприклад IsExcellent() та IsFailing().
Клас StudentGroup відповідає вже не за одного студента, а за групу. Він зберігає список студентів, додає і видаляє студентів, шукає їх, формує список відмінників, рахує середній бал групи та зберігає/завантажує дані з JSON-файлу.
Клас GradeJournal відповідає тільки за оцінки студента. Він зберігає оцінки по предметах у словнику та вміє рахувати середній бал.
Клас Program відповідає за меню та взаємодію з користувачем через консоль. Саме там користувач обирає дії: додати студента, видалити, знайти, редагувати, переглянути статистику, зберегти або завантажити дані .
Таким чином, кожен клас має свою окрему роль і не виконує зайвих обов’язків.

7. Які труднощі виникали під час проєктування архітектури? Як вирішували?
Під час виконання роботи виникло кілька труднощів.
Першою складністю було правильно розподілити логіку між класами. Спочатку могло здатися, що всі методи можна розмістити в одному класі, але це зробило б код незручним. Тому логіку було розділено: студент зберігається у класі Student, група — у StudentGroup, оцінки — у GradeJournal, а меню — у Program.
Другою складністю була валідація даних. Потрібно було зробити так, щоб користувач не міг створити студента з порожнім ПІБ, неправильним email або некоректним номером залікової книжки. Це було вирішено через властивості з повною реалізацією та винятки.
Третьою складністю було збереження списку студентів у файл. Для цього було використано JSON-серіалізацію та допоміжний внутрішній клас GroupDto, який дозволяє зручно зберігати назву групи, спеціальність, курс і список студентів.
Ще однією складністю було редагування студента. Щоб не змінювати дані напряму без перевірки, у програмі створюється оновлений об’єкт студента, після чого старий запис видаляється, а новий додається до групи .

8. Що нового ви дізналися під час виконання цієї роботи?
Під час виконання цієї практичної роботи я краще зрозуміла принцип інкапсуляції та навчилася застосовувати його в реальному коді. Я побачила, як приватні поля, властивості та методи допомагають захищати дані від неправильного використання.
Також я навчилася використовувати різні види властивостей: авто-властивості, властивості з повною реалізацією, read-only властивості, властивості з приватним setter та required-властивості.
Крім цього, я краще розібралася з валідацією даних у C#: перевіркою рядків, числових значень, email та дат. Також було корисно попрацювати з колекціями, зокрема зі списками List та словником Dictionary<string, double>.
Окремо я отримала досвід роботи із JSON-файлами для збереження та завантаження даних. Також під час виконання роботи я навчилася працювати з Git: створювати репозиторій, робити коміти, створювати гілки та відправляти роботу на GitHub.
