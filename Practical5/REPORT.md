# REPORT.md
1. Що таке наслідування? Які переваги та недоліки цього механізму?
Наслідування — це механізм об’єктно-орієнтованого програмування, який дозволяє створювати нові класи на основі вже існуючих. Похідний клас автоматично отримує поля, властивості та методи базового класу й може розширювати або перевизначати їх.
У даній практичній роботі базовим класом є Person, від якого успадковується клас Student, а вже від Student — класи:
* ExcellentStudent
* ForeignStudent
* WorkingStudent
* GraduateStudent
Також було реалізовано окрему ієрархію транспортних засобів:
* Vehicle
* Car
* Bus
* Truck 
Переваги наслідування:
* повторне використання коду;
* зменшення дублювання;
* можливість реалізації поліморфізму;
* зручне розширення функціоналу програми.
Недоліки:
* складніша структура проєкту;
* залежність дочірніх класів від базового;
* при неправильному проєктуванні виникає надмірна вкладеність класів.

2. Наведіть приклади використання base у вашому коді
Ключове слово base використовувалося для виклику конструктора базового класу та його методів.
Виклик конструктора базового класу:
public Student(...) 
    : base(fullName, dateOfBirth, personalEmail, notes)
public ForeignStudent(...) 
    : base(fullName, dateOfBirth, personalEmail, enrollmentDate, recordBookNumber, notes)
public Car(...) 
    : base(model, year, plateNumber)
Використання base.GetInfo():
return base.GetInfo() +
       $"Країна: {Country}\n";
return base.GetInfo() +
       $"Тип: Автобус\n";
Таким чином дочірні класи повторно використовують логіку базових класів. 

3. Поясніть різницю між virtual / override та new
Ключове слово  virtual  дозволяє оголосити метод, який можна перевизначати у дочірніх класах.
Ключове слово  override  використовується у похідному класі для зміни реалізації методу базового класу.
Наприклад:
public virtual string GetInfo()
та:
public override string GetInfo()
У програмі метод  GetInfo()  перевизначається у класах:
*  Student 
*  ExcellentStudent 
*  ForeignStudent 
*  WorkingStudent 
*  GraduateStudent 
*  Car 
*  Bus 
*  Truck 
Ключове слово  new  використовується для приховування методу базового класу, але в цій практичній роботі воно не застосовувалося, оскільки було використано саме поліморфізм через  virtual/override . 

4. Для чого потрібні абстрактні класи? Де ви їх використали?
Абстрактні класи використовуються для створення базової структури, яку не можна створити як окремий об’єкт, але від якої можуть успадковуватися інші класи.
У роботі було використано два абстрактних класи:
*  UniversityMember 
*  Vehicle 
Абстрактний клас  UniversityMember 
public abstract class UniversityMember : Person
Містить:
* абстрактний метод  CalculateScholarship() ;
* віртуальний метод  Enroll() .
Абстрактний клас  Vehicle  
public abstract class Vehicle
Містить:
* абстрактний метод  CalculateServiceCost() .
Абстрактні класи дозволили реалізувати спільну поведінку для всіх похідних класів. 

5. Навіщо використовують sealed класи та методи?
Ключове слово  sealed  забороняє подальше наслідування класу.
У роботі sealed-класом було зроблено:
public sealed class GraduateStudent : Student
Це означає, що від класу  GraduateStudent  більше не можна створювати дочірні класи.
 sealed  використовують:
* для захисту логіки класу;
* для обмеження ієрархії;
* для підвищення безпеки архітектури програми. 

6. Як ви реалізували поліморфізм у системі студентів?
Поліморфізм було реалізовано через базовий клас  UniversityMember .
Колекцію студентської групи було змінено:
private List<UniversityMember> _members = new();
Це дозволило зберігати в одній колекції об’єкти різних типів:
*  Student 
*  ExcellentStudent 
*  ForeignStudent 
*  WorkingStudent 
*  GraduateStudent 
Для роботи з поліморфними об’єктами були реалізовані методи:
public void AddMember(UniversityMember member) 
public List<T> GetMembersByType<T>()
public decimal GetTotalScholarship()
Під час виконання програми автоматично викликається відповідна реалізація методів дочірнього класу через  override . 

7. Які труднощі виникали при проектуванні ієрархії класів? Як їх вирішували?
Основною проблемою було переведення старої системи  List<Student>  на поліморфну колекцію  List<UniversityMember> . Через це виникали помилки:
* несумісність типів;
* проблеми з конструкторами;
* використання старих назв полів ( fullName ,  recordBookNumber ,  personalEmail );
* помилки при clone та operator overload.
Для вирішення проблем:
* було створено базовий клас  UniversityMember ;
* реалізовано  override  методів;
* усі старі поля були замінені на властивості:
  *  FullName 
  *  RecordBookNumber 
  *  PersonalEmail ;
* object initializer було замінено на виклики конструктора.
Також складністю було інтегрувати нове меню наслідування та окрему ієрархію транспортних засобів у вже існуючу систему. 

8. Що нового ви дізналися під час виконання цієї практичної роботи?
Під час виконання роботи було поглиблено знання з:
* наслідування;
* поліморфізму;
* abstract та sealed класів;
* virtual / override методів;
* generic-методів;
* побудови ієрархії класів;
* організації великого консольного меню;
* інтеграції нових модулів у вже існуючий проєкт.
Також було отримано практичний досвід роботи з:
* поліморфними колекціями;
* перевизначенням методів;
* абстрактними методами;
* реалізацією декількох незалежних ієрархій класів у межах одного проєкту.
